# Content Syncer

将年月目录（如 `25/01/`、`26/03/`）中的日志文件整理到 `collections` 目录，按类型分类管理，便于检索和维护。

---

## 一、目录结构

```
log-master/
├── 25/                    # 年份目录（两位年份，如 25 = 2025
│   └── 01/                # 月份目录（两位月份，如 01 = 1月
│       ├── 01.md          # 日志文件（两位日期，如 01 = 1日
│       └── 02.md
├── 26/
│   └── 03/
│       └── 15.md
└── collections/           # 分类整理目录
    ├── Css/               # CSS相关
    ├── Js/                # JavaScript相关
    ├── Git/               # Git相关
    ├── Software/          # 开发工具
    ├── Idea/              # 想法观点
    ├── Vue/               # Vue相关
    ├── React/             # React相关
    ├── RegExp/            # 正则表达式
    ├── Chrome/            # Chrome浏览器
    ├── Markdown/          # Markdown相关
    ├── Emoji/             # Emoji表情
    ├── Html/              # HTML相关
    └── Log/               # 同步记录
```

### 命名规则

| 层级 | 格式 | 示例 | 说明 |
|-----|------|------|------|
| 年份 | 两位数字 | `25` | 2025年 |
| 月份 | 两位数字 | `03` | 3月 |
| 日期 | 两位数字 | `15` | 15日 |

## 二、分类规则

### 分类列表

| 分类 | 内容类型 | 目标文件 |
|-----|---------|---------|
| **Css** | CSS特性、属性、CSS Tricks | `collections/Css/index.md` |
| **Js** | JavaScript API、语言特性 | `collections/Js/index.md` |
| **Git** | Git命令、版本控制技巧 | `collections/Git/index.md` |
| **Software** | 开发工具、网站、插件、配置 | `collections/Software/index.md` |
| **Idea** | 想法、观点、知识、设计原则 | `collections/Idea/index.md` |
| **Vue** | Vue框架相关知识 | `collections/Vue/index.md` |
| **React** | React框架相关知识 | `collections/React/index.md` |
| **RegExp** | 正则表达式语法、用法 | `collections/RegExp/index.md` |
| **Chrome** | Chrome浏览器、DevTools | `collections/Chrome/index.md` |
| **Markdown** | Markdown语法、使用技巧 | `collections/Markdown/index.md` |
| **Html** | HTML标签、语义化、SEO相关 | `collections/Html/index.md` |
| **Emoji** | 表情符号、符号组合 | `collections/Emoji/index.md` |

### 分类区分准则

| 分类 | 定义 | 示例 |
|-----|------|------|
| **Software** | 可直接使用的工具、库、配置 | "VSCode插件推荐"、"axios使用" |
| **Idea** | 观点、想法、知识、设计原则 | "设计模式"、"技术思考"、"架构理念" |
| **Js** | JavaScript语言特性和API | "Promise用法"、"Array方法" |
| **Vue/React** | 框架特定知识 | "Vue组件通信"、"React Hooks" |

### 边界条目分类指南

当条目处于多个分类的边界时，按以下优先级决策：

| 条目关键词 | 优先分类 | 说明 |
|-----------|---------|------|
| favicon | **Html** | 网站图标属于HTML范畴 |
| iframe | **Js** | window.parent/top/self 等DOM API |
| 性能/优化 | **Js** | 偏向JS实现方案 |
| 面试题 | **Js** | 除非明确框架问题，否则归入JS |
| 字体(font-family) | **Css** | 样式相关 |
| 颜色(color) | **Css** | 样式相关 |
| 布局(layout) | **Css** | 样式相关 |

**【核心原则】边界条目提醒用户：**

- 遇到边界条目时，先记录下来
- 继续处理其他确定分类的条目
- **处理完所有文件后**，向用户报告边界条目，由用户决定分类
- 用户确认后，按照用户指定的分类进行归类
- 等待用户回复期间，不得跳过或随意归类

## 三、详细工作流程（含多阶段检查）

### 【阶段1：前置检查与准备

#### 步骤1.1：环境验证与准备

| 检查项 | 验证内容 | 失败处理 |
|--------|----------|----------|
| **1.1.1** | 验证工作目录是否存在 | 若不存在，报告错误并终止 |
| **1.1.2** | 验证 `collections` 目录是否存在 | 若不存在，创建该目录及其所有子分类目录 |
| **1.1.3** | 验证各分类目录的 `index.md` 是否存在 | 若不存在，创建空的 `index.md` |
| **1.1.4** | 验证 `collections/Log/todo.md` 是否存在 | 若不存在，创建同步记录文件 |

**具体操作：

1. 列出当前工作目录，确认根目录存在
2. 检查 `collections` 目录是否存在
3. 为每个分类检查子目录：`Css/`、`Js/`、`Git/`、`Software/`、`Idea/`、`Vue/`、`React/`、`RegExp/`、`Chrome/`、`Markdown/`、`Html/`、`Emoji/`、`Log/`
4. 对每个目录，检查 `index.md` 是否存在，不存在则创建
5. 检查 `collections/Log/todo.md`，不存在则创建初始内容

#### 步骤1.2：确定同步范围

| 同步类型 | 触发方式 | 处理范围 |
|---------|---------|---------|
| 指定同步 | "同步26年3月" | 处理 `26/03/` 下所有文件 |
| 指定同步 | "整理25年" | 处理 `25/` 下所有月份目录 |
| 指定同步 | "同步所有" | 处理所有年份目录下的文件 |

**检查项1.2.1**：验证指定的年份/月份目录是否存在
**检查项1.2.2**：验证目录内是否有文件
**检查项1.2.3**：列出待处理文件清单，确认无误后继续

#### 步骤2.1：遍历日志文件遍历

|--------|----------|----------|
| **2.1.1** | 验证文件是否为 `.md` 文件 | 非 `.md` 文件标记为"非MD文件，后续特殊处理 |
| **2.1.2** | 验证文件大小是否为0 | 空文件直接跳过，记录在报告 |
| **2.1.3** | 验证文件是否可读 | 无法读取的文件记录错误，继续下一个 |

| 检查项 | 验证内容 | 处理方式 |
|--------|----------|----------|

| **2.2.2** | 验证内容是否仅含emoji | 仅emoji内容归入 Emoji 分类 |

### 【阶段3：内容解析与条目分割

#### 步骤3.1：按标题分割条目

**【最高优先级规则】** 以 `#`、`##`、`###` 标题为界，每个标题到下一个标题之间为一个完整条目。

**【核心处理规则 - 必须严格遵守】**

| 规则编号 | 规则内容 | 优先级 | 说明 |
|---------|---------|-------|------|
| **R0** | **禁止擅自修改标题** | ⭐⭐⭐⭐⭐⭐（最高） | **【绝对禁止】** 不得擅自修改、缩写、添加或改写原始标题文字内容，必须100%保持原样。仅允许移除数字前缀。 |
| **R1** | **保持链接完整性** | ⭐⭐⭐⭐⭐（最高） | 标题中的 Markdown 链接 `[文本](url)` 必须完整保留在同一行，不得分离为多行，不得修改链接格式 |
| **R2** | 清除数字前缀 | ⭐⭐⭐⭐ | 仅移除标题前的序号前缀（如 `1、`、`2、`、`13、`），保持标题其他内容不变 |
| **R3** | 保持原始格式 | ⭐⭐⭐ | 标题中的链接、文本内容保持原样，不做任何修改 |

#### 步骤3.2：解析检查清单

| 检查项 | 验证内容 | 失败处理 |
|--------|----------|----------|
| **3.2.1** | 识别所有标题（#、##、###） | 确保所有标题都被正确识别 |
| **3.2.2** | 验证每个条目标题数量 | 统计原始文件中的标题数量，与解析后的条目数量对比 |
| **3.2.3** | 验证无标题内容为空 | 标记并记录空条目 |
| **3.2.4** | 验证链接完整性检查 | 检查标题中的链接格式 `[文本](url)` 是否完整 |
| **3.2.5** | 保存原始标题对照 | 处理前后标题对比，确保未修改内容（除数字前缀外） |

#### 处理流程：
1. 读取文件全部内容
2. 按标题分割成独立条目
3. 跳过空条目（无标题或无实质内容）
4. **处理标题**：
   - 识别标题中的链接格式 `[文本](url)`
   - 保留链接的完整性，不破坏 `[文本](url)` 结构
   - 只移除数字前缀（如 `1、`、`2、`），不触碰链接部分
5. 保持原始内容格式不变
6. 逐条判断分类归属

#### 处理示例（正确 vs 错误）

| 原始内容 | 正确处理结果 | 错误处理结果（禁止） | 说明 |
|---------|-------------|---------------------|------|
| `### 13、[使用chrome浏览器直接调post接口](url)` | `### [使用chrome浏览器直接调post接口](url)` | `### 使用chrome浏览器直接调post接口\n> [url]` | 仅移除 `13、`，链接必须完整保留在标题内 |
| `### 5、[Vue.js技术揭秘](https://ustbhuangyi.github.io/vue-analysis/)` | `### [Vue.js技术揭秘](https://ustbhuangyi.github.io/vue-analysis/)` | `### Vue.js技术揭秘\n> [链接]` | 链接必须完整，不得拆分 |
| `### 2、禁止在对象实例上直接使用object.prototypes` | `### 禁止在对象实例上直接使用object.prototypes` | 同上（无链接） | 仅移除 `2、` |

#### 标题链接格式识别

必须完整识别并保留以下格式：
- `[链接文本](https://example.com)` - 标准链接
- `[链接文本](https://example.com/path?query=1)` - 带参数链接
- `[带空格的链接文本](https://example.com)` - 带空格的链接文本

**错误示例（禁止）**：
```markdown
### 13、使用chrome浏览器直接调post接口
> [使用chrome浏览器直接调post接口](url)
```

**正确示例（必须）**：
```markdown
### [使用chrome浏览器直接调post接口](url)
```

### 【阶段4：去重检查（三层检查）

#### 步骤4.1：去重检查清单

| 检查类型 | 检查内容 | 处理方式 |
|---------|---------|---------|
| **4.1.1 链接去重** | URL是否已存在于目标分类中 | 跳过重复，记录在报告 |
| **4.1.2 标题去重** | 清除前缀后的标题文字是否相同 | 跳过重复，记录在报告 |
| **4.1.3 内容去重** | 核心代码/描述是否重复 | 跳过重复，记录在报告 |

#### 去重操作流程：

1. **检查项4.1.1：链接去重
   - 提取条目中的所有URL
   - 在目标分类的 `index.md` 中搜索这些URL
   - 若已存在，标记为重复，跳过

2. **检查项4.1.2：标题去重
   - 使用处理后的标题（清除数字前缀后）
   - 在目标分类中搜索相同标题
   - 若已存在，标记为重复，跳过

3. **检查项4.1.3：内容去重
   - 提取条目核心内容（代码块、关键描述
   - 与目标分类中已有内容对比
   - 若高度相似，标记为重复，跳过

#### 步骤5.1：边界条目识别

3. 等待用户回复，确认分类方式
4. 用户确认后，再进行分类写入

## 四、强制要求（必须100%遵守）

> **【最高优先级】** collections 内必须包含全部日志文件内容，不能缺少任何条目。禁止创建非日志文件的原创内容。

> **【绝对禁止】** 不得擅自修改、缩写、添加或改写原始标题文字内容。必须100%保持原始标题原样，仅允许移除数字序号前缀（如 `1、`、`2、`）。

| 序号 | 规则 | 优先级 | 说明 |
|-----|------|-------|------|
| 0 | **禁止擅自修改标题** | ⭐⭐⭐⭐⭐⭐（最高） | **【绝对禁止】** 不得擅自修改、缩写、添加或改写原始标题文字内容，必须100%保持原样。仅允许移除数字前缀。 |
| 1 | **保持链接格式** | ⭐⭐⭐⭐⭐ | **【关键】** 标题中的 Markdown 链接 `[文本](url)` 必须保持在标题内，不得分离为多行，不得破坏链接结构 |
| 2 | **清除数字前缀** | ⭐⭐⭐⭐ | 标题前的 `1、`、`2、`、`3、` 等序号前缀应去除，使标题更简洁 |
| 3 | **保持原始格式** | ⭐⭐⭐ | 严格保持原始内容的格式不变，包括空行数量、代码块格式、引用格式等 |
| 4 | **禁止跳过条目** | ⭐⭐⭐⭐⭐ | **【强制】** 所有条目必须归入某个分类，不得以"不确定分类"为由跳过。无法确定时按边界条目指南处理。 |
| 5 | 单一分类 | ⭐⭐⭐ | 每个条目只能归入一个分类 |
| 6 | 完整报告 | ⭐⭐ | 同步完成后必须向用户报告新增、已存在、无法分类的内容 |
| 7 | 用户确认 | ⭐⭐ | 无法分类的内容必须等待用户确认后再处理 |
| 8 | **前置检查** | ⭐⭐⭐⭐⭐ | 每个阶段开始前必须完成前置检查 |
| 9 | **写入前备份** | ⭐⭐⭐⭐ | 修改任何文件前必须先备份 |
| 10 | **写入后验证** | ⭐⭐⭐⭐ | 每次写入后必须立即验证 |

**优先级说明**：
- 规则0（禁止擅自修改标题）优先级最高，任何情况下都不得违反
- 规则1（保持链接格式）在任何情况下都优先于规则2和规则3
- 规则2（清除数字前缀）与规则3（保持原始格式）冲突时，以规则2为准
- **所有规则都必须遵守，不得有任何例外

## 五、创建新分类

当内容无法归入现有分类时：
1. 向用户报告无法分类的具体内容
2. 等待用户确认分类方式或创建新分类
3. 创建新分类步骤：
   - 在 `collections/` 下创建文件夹（如 `NewCat/`）
   - 创建 `index.md` 文件
   - 更新本 skill 的分类规则

## 六、错误处理与回退机制

### 6.1 错误类型与处理

| 错误类型 | 检测方式 | 处理方式 |
|---------|---------|---------|
| **文件读取失败 | 尝试读取文件时捕获异常 | 记录错误，继续处理下一个文件 |
| **目录不存在** | 检查目录存在性 | 创建目录 |
| **文件写入失败** | 尝试写入时捕获异常 | 使用备份恢复，报告错误 |
| **内容解析错误 | 验证发现内容异常 | 跳过该条目，记录错误，继续 |
| **链接格式破坏** | 处理前后对比发现 | 立即停止，使用备份恢复 |

### 6.2 回退机制

**回退触发条件：
1. 发现内容可能丢失
2. 链接格式被意外破坏
3. 标题被意外修改
4. 用户要求回退

**回退操作步骤**：
1. 立即停止所有操作
2. 使用备份恢复受影响的文件
3. 报告回退操作
4. 分析错误原因
5. 重新开始处理

### 6.3 逐步验证策略

每个关键操作后立即验证：
- 文件读取后验证内容完整
- 条目解析后验证标题未改
- 内容写入后验证成功写入
- 同步完成后验证完整性

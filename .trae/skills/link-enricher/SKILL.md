# Link Enricher

将简单的链接列表（如 `游水:` 开头的内容）格式化为带标题和描述的 Markdown 文件。自动访问每个链接获取网页内容，根据内容生成更贴切的标题和描述。

---

## 一、工作流程

### 阶段1：读取与解析原始文件

#### 步骤1.1：读取原始文件
- 读取指定的 Markdown 文件
- 识别内容中的链接和文本块

#### 步骤1.2：解析原始格式
原始格式示例：
```
游水:
https://example.com

游水:
https://example2.com

游水:
input::-webkit-inner-spin-button {
    -webkit-appearance: none;
```

解析规则：
- 每个 `游水:` 块为一个独立条目
- 块内内容可能是纯链接，也可能是代码块或其他内容

---

### 阶段2：格式化基础结构

#### 步骤2.1：初步格式化
将原始内容转换为标准 Markdown 格式：
- 链接条目 → `### [链接](url)`
- 代码块条目 → `### 描述标题` + 代码块

#### 步骤2.2：初步格式验证
- 检查链接格式是否正确
- 检查代码块是否完整
- 验证空行分隔

---

### 阶段3：内容获取与增强（核心）

#### 步骤3.1：访问链接获取内容
- 使用 WebFetch 工具访问每个链接
- 获取网页的标题、描述、主要内容

#### 步骤3.2：生成贴切标题
根据获取的内容生成合适的标题：
- 优先使用网页的 `<title>` 标签内容
- 若没有标题，从内容中提取关键信息
- 保持简洁但有信息性
- 如果链接访问失败，使用链接URL作为标题

#### 步骤3.3：生成描述
- 提取网页的元描述（meta description）
- 或从内容中概括出简短描述
- 描述使用 `>` 引用格式
- 如果无法获取描述或链接访问失败，可以省略描述部分

#### 步骤3.4：非链接内容处理
对于不是链接的代码块或文本：
- 生成描述性标题
- 保持代码块完整性
- 添加适当的语言标识（如 ````css`）

#### 步骤3.5：混合内容处理
处理同时包含链接和代码块的条目：
- 链接作为标题，代码块保持原样
- 支持链接 + 描述 + 代码块的组合形式

---

### 阶段4：写入与验证

#### 步骤4.1：写入文件
- 按条目顺序写入
- 每个条目之间用空行分隔
- 标题使用 `###` 格式

#### 步骤4.2：验证格式
- 检查标题格式
- 检查链接完整性
- 验证描述格式

---

## 二、格式规范

### 输出格式

#### 链接条目（带描述）
```markdown
### [贴切标题](https://example.com)

> 网页内容的简要描述
```

#### 链接条目（无描述）
```markdown
### [贴切标题](https://example.com)
```

#### 代码块条目
```markdown
### 功能描述标题

```css
input::-webkit-inner-spin-button {
    -webkit-appearance: none;
}
```
```

#### 链接加代码块条目
```markdown
### [工具名称](https://example.com)

```bash
$ command --option
```
```

#### 链接加描述加代码块条目
```markdown
### [文章标题](https://example.com)

> 文章内容的简要描述

```html
<p>示例代码</p>
```
```

### 标题规则
- 标题应该准确描述链接内容
- 不要使用过于泛化的标题
- 可以包含中英文
- 如果无法获取网页标题，使用URL作为标题

### 描述规则
- 描述应该简洁（1-2句话）
- 概括链接的核心内容
- 使用引用格式 `>` 开头
- 描述部分可以省略（当无法获取或内容为空时）

## 三、处理示例

### 示例1：原始内容（链接）
```
游水:
https://yuanchuan.dev/polygon-shapes
```

### 处理后
```markdown
### [CSS Doodle clip-path 多边形形状与数学函数](https://yuanchuan.dev/polygon-shapes)

> 通过 CSS Doodle 的 @shape() 函数和 clip-path 属性，使用三角函数创建各种多边形、面具、动物等有趣形状
```

### 示例2：原始内容（代码块）
```
游水:
input::-webkit-inner-spin-button {
    -webkit-appearance: none;
```

### 处理后
```markdown
### 隐藏-spin-button样式

```css
input::-webkit-inner-spin-button {
    -webkit-appearance: none;
}
```
```

### 示例3：原始内容（链接加代码块）
```
游水:
https://nodejs.cn/api/cli/heapsnapshot_signal_signal.html

$ node --heapsnapshot-signal=SIGUSR2 index.js
```

### 处理后
```markdown
### [node执行的内存快照](https://nodejs.cn/api/cli/heapsnapshot_signal_signal.html)

```bash
$ node --heapsnapshot-signal=SIGUSR2 index.js
```
```

### 示例4：原始内容（纯文本标题加代码块）
```
游水:
字体推荐

@import url('https://fonts.bunny.net/css?family=inter');
```

### 处理后
```markdown
### 字体推荐

```css
@import url('https://fonts.bunny.net/css?family=inter');
```
```

## 五、注意事项

| 序号 | 检查项 | 说明 |
|-----|--------|------|
| 0 | **保持原始内容** | 不得修改原始文件内容，只做格式化和增强 |
| 1 | **链接完整性** | 确保链接在处理后仍然可用，不破坏 Markdown 链接格式 |
| 2 | **代码块保持** | 代码块必须完整保留，不得省略或修改代码内容 |
| 3 | **空行分隔** | 条目之间必须用空行分隔，保持良好的可读性 |
| 4 | **访问验证** | 访问链接时注意处理失败情况，设置合理的超时时间 |
| 5 | **标题准确性** | 标题应该准确反映内容，优先使用网页 `<title>` |
| 6 | **描述简洁性** | 描述要简洁有信息性，控制在 1-2 句话 |
| 7 | **网络稳定性** | 批量处理时注意网络状况，避免频繁请求 |
| 8 | **内容多样性** | 支持链接、代码块、混合内容等多种格式 |

## 六、错误处理

### 链接访问失败
- 当无法访问链接时，使用链接本身作为标题
- 不添加描述或使用简单描述（如"无法获取页面内容"）
- 记录失败的链接到日志，便于后续处理
- 继续处理其他条目，不受影响

### 内容解析失败
- 当无法解析网页内容时，使用默认格式（链接作为标题）
- 保持链接可用，不影响其他条目
- 跳过描述生成，保持条目简洁

### 非链接内容
- 对于代码块或其他非链接内容，生成描述性标题
- 保持内容完整性，不修改代码逻辑
- 自动识别代码语言并添加合适的语法标识

### 网络超时处理
- 设置合理的请求超时时间（建议 10-15 秒）
- 超时后使用链接 URL 作为标题，跳过描述
- 记录超时链接，便于后续手动处理

---

## 七、最佳实践

### 批量处理建议
- 建议每次处理不超过 20 个链接，避免网络压力
- 对于大量链接的文件，可以分批处理
- 处理前备份原始文件，防止意外情况

### 描述优化技巧
- 描述应突出链接的核心价值和用途
- 避免过于技术化的术语，保持通俗易懂
- 对于技术文章，可以简要说明核心技术点

### 标题优化技巧
- 标题应简洁明了，准确反映内容
- 中英文标题均可，保持自然流畅
- 避免过长的标题（建议不超过 50 个字符）

### 文件命名规范
- 原始文件建议使用 `YY/MM/DD.md` 格式（如 `26/05/18.md`）
- 处理后的文件保持原文件名，便于追溯
- 建议定期整理和归档处理后的文件
